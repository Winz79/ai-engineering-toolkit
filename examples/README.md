# Examples

Examples show how to apply the methodology without turning the generic toolkit into a product-specific process.

## Example Types

Use examples for:

- demonstrating artifact quality
- showing how one artifact leads to the next
- validating that templates and skills are practical
- exposing gaps in the methodology

Do not use examples for:

- inventing product strategy that was not provided
- encoding one project's assumptions into the generic methodology
- replacing the reusable templates
- skipping uncertainty by filling unknowns with plausible details

## Growth Pattern

Add examples one artifact at a time:

1. Start with intake.
2. Add context discovery only when source context exists.
3. Add requirements only after assumptions are explicit.
4. Add impact, design, delivery, review, and release artifacts as separate slices.
5. Update this index whenever an example gains a new artifact.

## Quality Bar

Each example should:

- state what source context it uses
- distinguish confirmed facts from assumptions
- link the template or skill it demonstrates
- avoid private, sensitive, or invented business context
- remain small enough to review independently
