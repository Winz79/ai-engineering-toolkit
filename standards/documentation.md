# Documentation Standard

Documentation should help the next person understand, use, review, operate, or change the system.

## Principles

- Document verified behavior, not aspirations.
- Keep documentation close to the artifact it supports.
- Prefer concise, practical guidance over broad narration.
- Link related artifacts instead of duplicating large sections.
- Update existing documentation when that is clearer than adding a new file.
- Preserve the distinction between generic methodology and project-specific examples.

## When To Update Documentation

Update documentation when a change affects:

- public behavior
- setup or usage
- configuration
- architecture or boundaries
- security or privacy expectations
- operational procedures
- testing or verification
- release or rollback behavior

## Minimum Useful Content

A good documentation update states:

- what changed
- who needs to know
- how to use or verify it
- what assumptions or limitations remain
- where related decisions or artifacts live

## Avoid

- documenting behavior that has not been implemented or verified
- repeating the same information across many files
- burying important operational details in long prose
- mixing example-specific facts into generic methodology
- leaving stale links or contradictory guidance
